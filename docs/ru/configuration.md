# Конфигурация

Настройки задаются классом `Settings` (`src/common/config/app_config.py`) на базе
pydantic-settings. Значения переопределяются переменными окружения с префиксом `DVD_` или через
файл `.env`. Имя переменной — `DVD_` плюс имя поля в верхнем регистре (например, поле
`vector_size` соответствует `DVD_VECTOR_SIZE`).

Поля-списки (`languages`, `allowed_extensions`) задаются в окружении в формате JSON, например
`DVD_ALLOWED_EXTENSIONS='[".docx",".txt",".md"]'`.

У каждого поля есть дефолт в коде, поэтому приложение запускается и вовсе без `.env`. В репозитории
два файла-примера: **`.env.example`** — сетевые связи сервиса, то, что обычно переопределяют
(локальные адреса Ollama/Qdrant/Redis, включение Kafka); **`.env.full.example`** — полный
справочник, зеркалирующий дефолты. Канонический перечень переменных — эта страница: при
добавлении настройки обновляйте таблицу здесь и `.env.full.example` вместе.

## Переменные

### Ollama

| Переменная | По умолчанию | Описание |
|------------|--------------|----------|
| `DVD_OLLAMA_BASE` | `http://a.dgx:11434` | адрес Ollama |
| `DVD_OLLAMA_MODEL` | `gpt-oss:20b` | LLM для разметки, мерджа, тегов, версии |
| `DVD_OLLAMA_EMBED_MODEL` | `bge-m3` | эмбеддинг-модель для `DVD_EMBEDDINGS_PROVIDER=ollama` |
| `DVD_OLLAMA_NUM_CTX` | `16384` | размер контекста модели |
| `DVD_OLLAMA_NUM_PREDICT` | `8192` | потолок токенов ответа |
| `DVD_OLLAMA_TIMEOUT` | `600.0` | таймаут запроса, сек |

### Векторизатор (провайдер эмбеддингов)

Векторизатор по умолчанию — GPU-сервис **giga-vectorizer** (отдельный репозиторий) с
OpenAI-совместимым `POST /v1/embeddings` и расширением `prompt`: документы векторизуются без
инструкции-префикса, запросы — с ней (Giga-Embeddings-instruct — асимметричная модель). Ollama
(`bge-m3`) остаётся резервным провайдером для окружений без GPU.

| Переменная | По умолчанию | Описание |
|------------|--------------|----------|
| `DVD_EMBEDDINGS_PROVIDER` | `giga` | `giga` — GPU-сервис (2048-d) / `ollama` — резерв через `/api/embed` (1024-d) |
| `DVD_EMBEDDINGS_URL` | `http://localhost:8001` | адрес giga-vectorizer |
| `DVD_EMBEDDINGS_MODEL` | `ai-sage/Giga-Embeddings-instruct` | модель, которую обслуживает векторизатор (сервис проверяет совпадение) |
| `DVD_EMBEDDINGS_QUERY_PROMPT` | `Instruct: Дан вопрос, необходимо найти абзац текста с ответом\nQuery: ` | инструкция-префикс для эмбеддинга запросов |
| `DVD_EMBEDDINGS_TIMEOUT` | `600.0` | таймаут запроса, сек |

Смена провайдера меняет векторное пространство: скорректируйте `DVD_VECTOR_SIZE`. При включённом
namespacing сервис будет использовать новую физическую коллекцию Qdrant, и документы нужно заново
загрузить в это новое пространство (эмбеддинги разных моделей несравнимы).

### Qdrant

| Переменная | По умолчанию | Описание |
|------------|--------------|----------|
| `DVD_QDRANT_URL` | `http://localhost:6333` | адрес Qdrant |
| `DVD_QDRANT_API_KEY` | пусто | ключ API (если требуется) |
| `DVD_QDRANT_COLLECTION` | `documents` | **базовое** имя коллекции (см. «Пространства коллекций») |
| `DVD_VECTOR_SIZE` | `2048` | **только резервное значение** — реальная размерность определяется опросом активного векторизатора при старте и перезаписывает это значение; используется как есть, только если векторизатор недоступен на старте (giga = 2048, bge-m3 = 1024) |
| `DVD_EMBED_BATCH` | `32` | размер батча при векторизации |
| `DVD_COLLECTION_NAMESPACING` | `true` | отдельная физическая коллекция под каждое векторное пространство (см. ниже) |

#### Пространства коллекций (namespacing)

При `DVD_COLLECTION_NAMESPACING=true` (по умолчанию) **физическая** коллекция Qdrant — это не
`DVD_QDRANT_COLLECTION` как есть, а имя, собранное из базового имени, модели эмбеддингов и
размерности:

```
{base}__{model_slug}_{dim}
documents__giga_embeddings_instruct_2048     # giga / 2048
documents__bge_m3_1024                        # резервный ollama / 1024
```

На старте сервис **опрашивает активный векторизатор** и берёт его реальную размерность эмбеддинга в
`vector_size`, затем создаёт коллекцию, на которую указывает **текущий** конфиг, если её ещё нет. Так
как имя кодирует векторное пространство (и авто-определённую размерность), смена провайдера/модели/
размерности попадает в **новую** коллекцию, а прежняя остаётся нетронутой — переключение провайдера
(или откат) никогда не перезаписывает и не смешивает пространства, и не нужен ручной «удалить +
переиндексировать».
Документы в новое пространство всё равно нужно залить заново (эмбеддинги разных моделей
несравнимы), но старое пространство остаётся доступным.

Реестр документов в Redis (хэши дедупа, наборы версий, сводки документов) неймспейсится так же
(`dvd:{effective_collection}:…`), поэтому свежее пространство получает и чистый реестр — иначе
проверка дубликатов ошибочно отклоняла бы повторную загрузку документов, которые есть только в
**старом** пространстве.

`DVD_COLLECTION_NAMESPACING=false` — использовать `DVD_QDRANT_COLLECTION` как есть (прежнее
поведение, префикс реестра `dvd:`). В этом режиме, если размерность существующей коллекции не
совпадает с `DVD_VECTOR_SIZE`, старт **падает с явной ошибкой**, а не пишет несовместимые векторы.
Коллекция обученных паттернов ссылок (`DVD_REF_PATTERN_COLLECTION`) не зависит от модели и никогда
не неймспейсится.

### MinIO (исходные файлы документов)

Каждая загрузка/PATCH/PUT на `/documents` и `/user-documents` сохраняет исходный файл в MinIO,
который работает за закрытым сетевым контуром — в ответах API наружу отдаётся только ссылка на
сам сервис (`GET /documents/{name}/source`, `GET /user-documents/{name}/source`), который
проксирует скачивание. Сохранение — fail-closed: если MinIO недоступен, запрос завершается
ошибкой до постановки в очередь на индексацию. Объекты ключуются по хешу содержимого
(`{content_hash}{расширение}`), поэтому повторная загрузка идентичного контента переиспользует тот
же объект. Два бакета разделяют общий корпус и пользовательские индексы; удаление
версии/документа/целого пользовательского индекса удаляет и соответствующие объекты (best-effort —
ошибка MinIO здесь только логируется и не блокирует удаление в Qdrant/Redis).

| Переменная | По умолчанию | Описание |
|------------|--------------|----------|
| `DVD_MINIO_ENDPOINT` | `localhost:9000` | адрес MinIO/S3 (`host:port`) |
| `DVD_MINIO_ACCESS_KEY` | `minioadmin` | access key |
| `DVD_MINIO_SECRET_KEY` | `minioadmin` | secret key |
| `DVD_MINIO_SECURE` | `false` | использовать HTTPS для подключения к MinIO |
| `DVD_MINIO_BUCKET_DOCUMENTS` | `dvd-documents` | бакет общего/обычного корпуса документов |
| `DVD_MINIO_BUCKET_USER_DOCUMENTS` | `dvd-user-documents` | бакет всех пользовательских индексов |

### Kafka (события жизненного цикла документов)

Публикация опциональна и выключена, пока не задан `DVD_KAFKA_BOOTSTRAP_SERVERS`. Изменения
жизненного цикла документов добавляются в надёжный outbox в Redis и доставляются в топик
`document.events` фоновым публикатором (фреймворк
[otteroad](https://github.com/IDUclub/otteroad): AVRO + Schema Registry). События, которые
раз за разом не удаётся отправить, переносятся в dead-letter-список и не блокируют очередь.
Типы событий:

| Событие | Когда | Payload |
|---------|-------|---------|
| `DocumentProcessed` | первичная загрузка документа (`POST /documents`) | `document_name` |
| `DocumentUpdated` | дельта-обновление или полная перезагрузка (`PATCH`/`PUT /documents/{name}`) | `document_name`, `version` |
| `DocumentDeleted` | удаление документа или одной версии (`DELETE /documents/{name}`) | `document_name`, `versions_removed`, `document_removed` |

`PUT`-перезагрузка объявляется одним `DocumentUpdated` (без промежуточного `DocumentDeleted`);
перезагрузка ещё не сохранённого документа объявляется как `DocumentProcessed`.

| Переменная | По умолчанию | Описание |
|------------|--------------|----------|
| `DVD_KAFKA_BOOTSTRAP_SERVERS` | пусто | брокеры Kafka (`host:port[,host:port]`); пусто = публикация выключена |
| `DVD_KAFKA_SCHEMA_REGISTRY_URL` | `https://schema-registry.next.idulab.ru` | адрес AVRO Schema Registry (дефолт — реестр контура IDU) |
| `DVD_KAFKA_CLIENT_ID` | `idu-dvd` | идентификатор клиента в логах брокера |
| `DVD_KAFKA_OUTBOX_KEY` | `dvd:kafka:outbox` | Redis-список с ожидающими событиями |
| `DVD_KAFKA_DEAD_LETTER_KEY` | `dvd:kafka:outbox:dead` | Redis-список событий, исчерпавших ретраи |
| `DVD_KAFKA_POLL_INTERVAL` | `1.0` | пауза между проверками outbox при простое, сек |
| `DVD_KAFKA_RETRY_INTERVAL` | `5.0` | пауза после неудачной отправки, сек |
| `DVD_KAFKA_MAX_ATTEMPTS` | `10` | число попыток отправки до переноса в dead-letter |

## Важные замечания

- `DVD_VECTOR_SIZE` определяется автоматически на старте опросом активного векторизатора, поэтому
  обычно сам совпадает с моделью (Giga-Embeddings-instruct — 2048, `bge-m3` — 1024); заданное
  значение — лишь резерв на случай, когда векторизатор недоступен при старте. Размерность фиксируется
  при создании коллекции Qdrant и не меняется без её пересоздания. При включённом namespacing (по
  умолчанию) смена размерности просто создаёт новую коллекцию; при выключенном — переиндексируйте
  существующую (или включите namespacing).
- `DVD_PARTITION_STRATEGY` влияет только на разбор форматов, отличных от `.docx`; `.docx`
  разбирается через `partition_docx` без учёта стратегии.
- По умолчанию адрес Ollama и LLM указывают на стенд (`a.dgx`, `gpt-oss:20b`). Для локального
  запуска переопределите их в `.env`.

## Рекомендации по моделям

### Только CPU (CI, без GPU)

- LLM: `qwen2.5:3b-instruct` — достаточно компактна для CPU-инференса; интеграционный набор
  тестов с ней проходит (структурированный вывод в Ollama ограничен грамматикой схемы, поэтому
  валидность JSON не зависит от размера модели). Используется в CI-workflow интеграционных тестов.
  Качество разметки ниже, чем у моделей 7B+ — для тестов достаточно, для реальных корпусов не
  рекомендуется.
- Эмбеддинги: провайдер `ollama` с `bge-m3` (и так работает на CPU); giga-vectorizer работает
  только с CUDA и без GPU не стартует.

```
DVD_EMBEDDINGS_PROVIDER=ollama
DVD_OLLAMA_EMBED_MODEL=bge-m3
DVD_VECTOR_SIZE=1024
```

### Локальный запуск на скромном GPU

- LLM: `qwen2.5:7b-instruct` (хороший русский и устойчивый структурированный вывод, помещается
  на 8 ГБ видеопамяти).
- Эмбеддинги: giga-vectorizer (Giga-Embeddings-instruct, 2048) — дефолт; на картах 8 ГБ модели
  тесно, поэтому на единственном небольшом GPU, разделяемом с LLM, лучше резервный вариант
  `ollama` (`bge-m3`, 1024).

```
DVD_OLLAMA_BASE=http://localhost:11434
DVD_OLLAMA_MODEL=qwen2.5:7b-instruct
DVD_EMBEDDINGS_URL=http://localhost:8001
```

### Сервер с несколькими Tesla V100

Tesla V100 (архитектура Volta) надёжно работает в FP16; быстрые int4-ядра (AWQ, GPTQ-marlin),
FP8 и Flash-Attention 2 не поддерживаются. Используйте FP16 (при переходе на vLLM) или
GGUF-квантование (при работе через Ollama).

- LLM: `qwen2.5:14b-instruct` как рабочий вариант (заметно качественнее 7B, помещается с запасом),
  при необходимости `qwen2.5:32b-instruct` для большего качества и наличии 32 ГБ-карт.
- Эмбеддинги: giga-vectorizer (Giga-Embeddings-instruct, FP16 — дружественно Volta) на выделенной GPU.
- Раскладка на три карты: LLM с тензорным параллелизмом на двух GPU (для 14B допустим
  параллелизм 2; параллелизм 3 не используется из-за неделимости числа голов), эмбеддинги на
  третьей GPU.
- Основной резерв ускорения — параллельная обработка окон конвейера и батчинг запросов к модели
  (через vLLM или через `OLLAMA_NUM_PARALLEL` и конкурентные запросы).
