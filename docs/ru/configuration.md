# Конфигурация

Настройки задаются классом `Settings` (`src/common/config/app_config.py`) на базе
pydantic-settings. Значения переопределяются переменными окружения с префиксом `DVD_` или через
файл `.env`. Имя переменной — `DVD_` плюс имя поля в верхнем регистре (например, поле
`vector_size` соответствует `DVD_VECTOR_SIZE`).

Поля-списки (`languages`, `allowed_extensions`) задаются в окружении в формате JSON, например
`DVD_ALLOWED_EXTENSIONS='[".docx",".txt",".md"]'`.

У каждого поля есть дефолт в коде, поэтому приложение запускается и вовсе без `.env`. В репозитории
два файла-примера: **`.env.example`** — сетевые связи сервиса, то, что обычно переопределяют
(локальные адреса Ollama/Qdrant/Redis, включение Kafka); **`.env.full.example`** — полный
справочник, зеркалирующий дефолты. Канонический перечень переменных — эта страница: при
добавлении настройки обновляйте таблицу здесь и `.env.full.example` вместе.

### Kafka (события жизненного цикла документов)

Публикация опциональна и выключена, пока не задан `DVD_KAFKA_BOOTSTRAP_SERVERS`. Изменения
жизненного цикла документов добавляются в надёжный outbox в Redis и доставляются в топик
`document.events` фоновым публикатором (фреймворк
[otteroad](https://github.com/IDUclub/otteroad): AVRO + Schema Registry). События, которые
раз за разом не удаётся отправить, переносятся в dead-letter-список и не блокируют очередь.
Типы событий:

| Событие | Когда | Payload |
|---------|-------|---------|
| `DocumentProcessed` | первичная загрузка документа (`POST /documents`) | `document_name` |
| `DocumentUpdated` | дельта-обновление или полная перезагрузка (`PATCH`/`PUT /documents/{name}`) | `document_name`, `version` |
| `DocumentDeleted` | удаление документа или одной версии (`DELETE /documents/{name}`) | `document_name`, `versions_removed`, `document_removed` |

`PUT`-перезагрузка объявляется одним `DocumentUpdated` (без промежуточного `DocumentDeleted`);
перезагрузка ещё не сохранённого документа объявляется как `DocumentProcessed`.

| Переменная | По умолчанию | Описание |
|------------|--------------|----------|
| `DVD_KAFKA_BOOTSTRAP_SERVERS` | пусто | брокеры Kafka (`host:port[,host:port]`); пусто = публикация выключена |
| `DVD_KAFKA_SCHEMA_REGISTRY_URL` | `https://schema-registry.next.idulab.ru` | адрес AVRO Schema Registry (дефолт — реестр контура IDU) |
| `DVD_KAFKA_CLIENT_ID` | `idu-dvd` | идентификатор клиента в логах брокера |
| `DVD_KAFKA_OUTBOX_KEY` | `dvd:kafka:outbox` | Redis-список с ожидающими событиями |
| `DVD_KAFKA_DEAD_LETTER_KEY` | `dvd:kafka:outbox:dead` | Redis-список событий, исчерпавших ретраи |
| `DVD_KAFKA_POLL_INTERVAL` | `1.0` | пауза между проверками outbox при простое, сек |
| `DVD_KAFKA_RETRY_INTERVAL` | `5.0` | пауза после неудачной отправки, сек |
| `DVD_KAFKA_MAX_ATTEMPTS` | `10` | число попыток отправки до переноса в dead-letter |

### Дефолты идентичности документа

Универсальные значения по умолчанию для кросс-сервисных полей payload, если загрузчик их не передал;
доменные потребители (например, MSI-TSIM) переопределяют их per-upload через поля формы / `external_ids`.

| Переменная | По умолчанию | Описание |
|------------|--------------|----------|
| `DVD_DEFAULT_DOC_TYPE` | `document` | `doc_type` по умолчанию (`document`/`regulation`/`article`/`book`/`web`/…) |
| `DVD_DEFAULT_CORPUS` | `default` | логический корпус/неймспейс по умолчанию |
| `DVD_DEFAULT_LANG` | пусто | код языка ISO-639 по умолчанию (нет = неизвестно) |

## Важные замечания

- `DVD_VECTOR_SIZE` обязан совпадать с размерностью выбранной эмбеддинг-модели. Для `bge-m3`
  это 1024. Размерность фиксируется при создании коллекции Qdrant и не меняется без её
  пересоздания.
- `DVD_PARTITION_STRATEGY` влияет только на разбор форматов, отличных от `.docx`; `.docx`
  разбирается через `partition_docx` без учёта стратегии.
- По умолчанию адрес Ollama и LLM указывают на стенд (`a.dgx`, `gpt-oss:20b`). Для локального
  запуска переопределите их в `.env`.

## Рекомендации по моделям

### Только CPU (CI, без GPU)

- LLM: `qwen2.5:3b-instruct` — достаточно компактна для CPU-инференса; интеграционный набор
  тестов с ней проходит (структурированный вывод в Ollama ограничен грамматикой схемы, поэтому
  валидность JSON не зависит от размера модели). Используется в CI-workflow интеграционных тестов.
  Качество разметки ниже, чем у моделей 7B+ — для тестов достаточно, для реальных корпусов не
  рекомендуется.
- Эмбеддинги: `bge-m3` (и так работает на CPU).

### Локальный запуск на скромном GPU

- LLM: `qwen2.5:7b-instruct` (хороший русский и устойчивый структурированный вывод, помещается
  на 8 ГБ видеопамяти).
- Эмбеддинги: `bge-m3` (мультиязычная, 1024).

```
DVD_OLLAMA_BASE=http://localhost:11434
DVD_OLLAMA_MODEL=qwen2.5:7b-instruct
DVD_OLLAMA_EMBED_MODEL=bge-m3
DVD_VECTOR_SIZE=1024
```

### Сервер с несколькими Tesla V100

Tesla V100 (архитектура Volta) надёжно работает в FP16; быстрые int4-ядра (AWQ, GPTQ-marlin),
FP8 и Flash-Attention 2 не поддерживаются. Используйте FP16 (при переходе на vLLM) или
GGUF-квантование (при работе через Ollama).

- LLM: `qwen2.5:14b-instruct` как рабочий вариант (заметно качественнее 7B, помещается с запасом),
  при необходимости `qwen2.5:32b-instruct` для большего качества и наличии 32 ГБ-карт.
- Эмбеддинги: `bge-m3` (без изменений).
- Раскладка на три карты: LLM с тензорным параллелизмом на двух GPU (для 14B допустим
  параллелизм 2; параллелизм 3 не используется из-за неделимости числа голов), эмбеддинги на
  третьей GPU.
- Основной резерв ускорения — параллельная обработка окон конвейера и батчинг запросов к модели
  (через vLLM или через `OLLAMA_NUM_PARALLEL` и конкурентные запросы).
